## 목표
- graphviz 문법을 이해해봅니다.
- dot 형식을 지원하는 그래프 생성도구를 설계해 봅니다.
## 요구사항

- 문법에 익숙해질 것.
- 최대한 같이 결정할 것
- 구조, 속성, 이름이 바뀌어도 동작해야합니다.
- 문제와 작동 방식을 요약해서 손으로 그려봅니다.
### 프로그래밍
- 패키지 -> 파일 -> 타입 -> 속성 단계의 구조를 그래프로 데이터를 넘기면, graphviz 출력 형식 문자열로 변환해야 합니다.
## 분석
먼저 무엇을 요구하는지 부터 살펴봤습니다.
![image](https://github.com/gxxhxx0224/Naver-BoostCamp/assets/80369805/fc4aee71-bf90-42c3-890c-b86b88a91187)
그림처럼 어떠한 입력을 받고서, 프로그래밍을 통해 graphviz 형식의 문자열로 만들어내야 한다고 요구사항을 해석했습니다.

### 입력
그렇다면 입력은 무엇이 되어야 하는가가 다음 고민이었습니다.

여러 패키지를 받을 수 있어야 하니까, 패키지의 배열이 적절하다고 생각을 모았습니다.
![image](https://github.com/gxxhxx0224/Naver-BoostCamp/assets/80369805/c06005c1-f898-485e-83cc-51fb038d3213)


패키지는 여러 파일을 가질 수 있으니 파일 배열을 가져야하며, 파일은 하나의 타입을 포함하니 1:1 매칭이라고 보았습니다.

이어서 타입은 여러 속성을 가질 수 있고, 속성은 속성이름과 그 타입에 대한 정보가 필요합니다.

### 변환
예시의 graphviz 코드를 보면 subgraph가 반복되어지는 것을 확인할 수 있었습니다.

따라서 작은 부분부터 자신을 표현할 수 있으면 상위 개념이 이를 바탕으로 또 자신을 표현하고, 나아가 subgraph, digraph를 생성하는 것이 구조적으로 아름다운 방식이 될 거라고 결론을 냈습니다.

### 과정
1. 제일 작은 단위인 속성부터 보면 이름; 형태입니다.
2. 두 번째는 파일-타입이며, 서브그래프, 타입명, (1.)의 표현들, 파일명;의 형태입니다.
3. 세 번째는 패키지이며, 서브그래프, 패키지명;, (2.)의 표현들의 형태입니다.
4. 마지막으로 모든 속성들에 대한 타입을 명시해 줍니다.
5. 이 모든 과정을 수행했다면 문자열로 변환해서 반환합니다.
위의 각 과정은 해당 컴포넌트에서 수행하도록 합니다.
